# 01 — Архитектурный план

Технический фундамент системы: стек, слои, границы, online/offline, deployment. Логику данных и модулей см. в [02 — Логический план](./02-logical.md); фичи — в [03 — Функциональный план](./03-functional.md).

## 1. Технологический стек

| Слой | Технология | Обоснование |
|---|---|---|
| **Клиент** | **Expo (React Native, TypeScript)** | Один код на iOS/Android; уже отскаффолжен в `learningFront`. Знаком команде. |
| **Навигация** | expo-router | File-based routing, deep links. |
| **Локальная БД** | **expo-sqlite + Drizzle ORM** | Offline-first источник правды для повторений и ответов. Типобезопасные миграции. |
| **SRS** | **ts-fsrs** | FSRS на клиенте, работает офлайн. |
| **Состояние** | Zustand + TanStack Query | Локальный стор + кэш серверных запросов с офлайн-персистом. |
| **Backend** | **FastAPI (Python)** | AI-оркестрация, скоринг, генерация, sync. Сильная сторона команды (ML/Python). |
| **Серверная БД** | **PostgreSQL** (managed, напр. Supabase Postgres) | Реляционные данные для аналитики прогресса; JSONB для гибких payload'ов. |
| **Auth** | Supabase Auth (или собственный JWT) | Вынесено из бизнес-логики; клиент хранит токен в SecureStore. |
| **AI** | **Claude API** (`claude-opus-4-8` / `claude-sonnet-4-6`) через FastAPI-gateway | Скоринг по рубрикам, генерация контента, ревью кода. Ключ только на сервере. |
| **STT (фаза 2)** | Whisper (сервер) или on-device STT | Для Speaking. Отложено за пределы MVP. |
| **TTS (фаза 2)** | Провайдер TTS | Для listening-материалов. |

**Выбор модели LLM:** скоринг продукции по рубрикам — `claude-opus-4-8` (качество оценки критично). Генерация карточек/дриллов и простые проверки — `claude-sonnet-4-6` (дешевле, быстрее). Модель выбирается per-rubric в реестре, а не хардкодится.

## 3. Ось связности: online / offline

**Принцип:** нет «оффлайн-версии» и «онлайн-версии». Есть один клиент, где каждая Activity помечена флагом `connectivity ∈ {offline, online}`.

### Что работает офлайн (без сети, локально)
- SRS-повторения (FSRS считается на клиенте).
- Просмотр материалов/заметок, уже синканных локально.
- Exam-дриллы с детерминированной проверкой (multiple choice с известным ответом).
- Написание эссе/ответа/кода — сохраняется в SQLite, ставится в очередь на скоринг.
- Статистика, прогресс, error-log.
- **Деградированный офлайн-fallback** для продукции: локальные эвристики (объём, структура абзацев, повторы, покрытие Academic Word List) дают мгновенный черновой сигнал до полного скоринга.

### Что требует сети (backend + LLM)
- AI-скоринг продукции по рубрикам.
- Генерация passages / карточек / дриллов (в т.ч. из PDF).
- Пересчёт адаптивного плана.
- Разговорная практика.
- Первичная диагностика уровня.
- Синхронизация.

### Мост offline → online: очередь отложенных задач

```
[Офлайн] Пишешь эссе
   → response сохраняется в SQLite
   → создаётся job {type: grade_writing, status: pending, input_ref}
[Появилась сеть]
   → sync выгружает pending-jobs в POST /jobs
   → FastAPI вызывает Claude по нужной рубрике
   → результат сохраняется в Postgres и возвращается
   → клиент обновляет UI, показывает разбор, кормит error-log
```

Детали схемы `job` и алгоритма sync — в [02 — Логический план](./02-logical.md#job-queue-и-синхронизация).

## 4. Границы систем и ответственность

| Компонент | Отвечает за | НЕ отвечает за |
|---|---|---|
| **Клиент (core-engine)** | Повторения, локальный event log, планирование FSRS, очередь jobs, рендеринг Activity | Вызовы LLM, хранение ключей, генерацию контента |
| **Клиент (модули)** | UI-рендереры своих Activity, локальные детерминированные проверки, импорт-UX | Рубрики/промпты (живут на бэке) |
| **FastAPI core** | Sync, обработка jobs, auth, AI-gateway | Предметную специфику |
| **FastAPI модули** | Рубрики, генераторы контента, доменные проверки | Общую инфраструктуру |
| **Claude API** | Оценку, генерацию, ревью | Хранение состояния |

## 5. Структура репозитория (суперпроект + сабмодули)

Репозиторий — **git-суперпроект** с двумя сабмодулями (отдельные репозитории). `docs/` — в суперпроекте.

```
/                         (D:\work\learning — суперпроект)
├── docs/                 ← этот раздел (источник правды)
├── learningFront/        ← САБМОДУЛЬ: клиент (Expo/RN), архитектура FSD
│   └── src/
│       ├── app/          FSD app-слой + expo-router роуты + composition root
│       ├── pages/        экраны
│       ├── widgets/      ActivityDispatcher, TodayQueue
│       ├── features/     по типам Activity (vocab-review, ielts-writing, …)
│       ├── entities/     activity, srs-card, response, module
│       └── shared/
│           ├── engine/   ← доменно-независимое ядро (Activity, реестр, FSRS, порты)
│           ├── ui/       ui-kit
│           ├── api/      http-клиент, sync-адаптер, SQLite LocalStore
│           ├── config/   env, константы
│           └── lib/      утилиты
└── learningBack/         ← САБМОДУЛЬ: backend (FastAPI, Python, uv)
    ├── core/             sync, jobs, auth, ai_gateway
    └── modules/
        ├── languages/    рубрики, генераторы
        └── ml/
```

Фронтенд организован по **Feature-Sliced Design** — детали и маппинг Praxis→FSD в [04 — Фронтенд (FSD)](./04-frontend-fsd.md). Ядро (ex «core-engine») живёт в `src/shared/engine`.

Backend — отдельное Python-окружение на **uv**, связан с клиентом только по HTTP-контракту. Инструменты фронта — обычный **npm** (без workspaces: единое приложение, общий код — FSD-слои, а не пакеты).

## 6. Deployment

| Часть | Среда | Способ |
|---|---|---|
| Мобильный клиент | iOS/Android | **EAS Build** (Expo Application Services); OTA-обновления через EAS Update |
| Backend | Контейнер | Docker → любой хостинг (Fly.io / Railway / VPS) |
| БД | Managed Postgres | Supabase / Neon |
| Секреты | Env | Ключ Claude, DB URL — только в окружении backend |

**Версионирование контракта:** клиент и backend согласуют версию API (`/v1/...`). Рубрики версионируются независимо (см. инвариант №6) — их можно менять без релиза приложения.

## 7. Нефункциональные требования

- **Приватность:** пользовательские тексты (эссе, ответы) уходят в Claude только для скоринга; политику хранения на бэке зафиксировать в отдельном документе перед публичным релизом.
- **Стоимость LLM:** кэш результатов генерации; выбор дешёвой модели где возможно; батчинг. Контроль через AI-gateway.
- **Производительность повторений:** офлайн, мгновенно (< 16 мс на планирование карточки), без сети.
- **Устойчивость sync:** идемпотентность jobs, last-write-wins на уровне записи (детали — [02](./02-logical.md#job-queue-и-синхронизация)).
